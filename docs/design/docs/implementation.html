OSCE Exam Simulator — Claude Code Implementation Plan#

Source BRD: OSCE Exam Simulator PRD v1.8 (12 April 2026) Target executor: Claude Code (Sonnet 4.6 / Opus 4.7) Plan prepared by: Oxolus Technologies

1. Executive Summary#

The BRD describes a chat-based OSCE exam simulator with four AI surfaces (case generation, standardized-patient simulation, scoring, readiness/Knowledge Bridge), a candidate-facing app, and a super-admin console. It is AI-heavy and orchestration-heavy rather than UI-heavy — most of the engineering work is in the AI module boundaries, structured outputs, state tracking, and token accounting, not the screens.

This plan recommends building it as a single Next.js 15 (App Router) application backed by Supabase (Postgres + Storage), with Drizzle ORM, Anthropic Claude as the primary LLM, OpenAI Whisper for STT, ElevenLabs for TTS, and Resend for transactional email. Hosting on Vercel.

Build sequence aligns to the BRD's three release phases. Each phase ends with a working, demoable slice rather than horizontal layers. Each AI module is a separate file with a versioned system prompt and a Zod schema for its structured output — this is non-negotiable because the BRD requires versioning, reliable rendering, and per-operation token tracking.

The biggest risks are not technical — they are clinical and economic: AI-generated X-rays are unreliable, AI-scored OSCEs need a defensible rubric to gain trust, and token costs at scale can spiral. Section 3 addresses each.

2. Critical Decisions & Risks (Read Before Coding)#

These decisions block clean execution. Resolve them with Teqplan / Eng. Mohammad Shihan before Phase 1.

2.1 X-ray and medical image generation#

The BRD requires X-ray images as a case asset (F4). Generating realistic, clinically plausible X-rays from a text-to-image model is unreliable — Flux, SDXL, and DALL-E produce images that look like X-rays but are anatomically wrong. This is a clinical-safety risk even in a simulator.

Three viable strategies, in order of recommendation:

  1. Curated library + tag matching. Build a library of ~200 properly licensed OSCE-suitable medical images (CC-licensed teaching collections, Radiopaedia educational use, etc.), tag each with specialization + condition + finding, and have the case generator select rather than generate. Add new images over time. Lowest risk, highest realism.
  2. Generation with explicit "simulated/non-diagnostic" watermark. Use Flux Pro with strong negative prompts and overlay a watermark. Acceptable as a fallback but introduces clinical risk.
  3. Skip image assets in Phase 1. Cases ship with text symptoms only. Re-introduce when library is ready.

Recommendation: Phase 1 ships with curated library (start with 30–50 images covering the launch specialization). Generation is a Phase 3 explore-only option.

2.2 Case voice recording#

The BRD lists "voice recording" as a case asset (F4). This is the simulated-patient voice clip the candidate listens to as part of the case stem (different from F5's live TTS during chat). Generate at case-creation time using ElevenLabs with a consistent voice ID per persona type (elderly male, middle-aged female, child, etc.). Cache in Supabase Storage keyed by case ID. Cost: ~$0.05–0.15 per case.

2.3 AI provider strategy#

The BRD is provider-agnostic. Recommendation:

  • Anthropic Claude Sonnet 4.6 for case generation, scoring, Knowledge Bridge (highest reasoning quality, JSON-mode reliable, best at staying in role).
  • Anthropic Claude Haiku 4.5 for simulation turn responses (cheaper, fast, sufficient for SP responses).
  • OpenAI Whisper for STT (best price/quality, no Anthropic equivalent yet).
  • ElevenLabs for TTS (significantly more natural than OpenAI/Azure for patient roleplay).

Abstract every call through a thin provider interface in lib/ai/providers/ so swapping later is mechanical, not architectural.

2.4 Scoring reliability#

The BRD requires AI scoring out of 240 with domain breakdown. AI scoring is notoriously inconsistent between runs. Mitigations the implementation must enforce:

  • Deterministic seed where supported (temperature: 0.0 on scoring calls).
  • Two-call scoring with reconciliation: run the scorer twice, compare deltas, flag if >15 points apart. Phase 2.
  • Rubric in the prompt, not the model. Each domain has an explicit 0–60 rubric stored in lib/ai/rubrics/, versioned. Changing rubric = new prompt version logged with the exam.
  • Server-side pass/fail computation. AI returns scores only; pass/fail is computed by comparing total_score >= pass_threshold in application code. AI cannot override the threshold.

2.5 Token cost ceiling#

Without a ceiling, a single user looping cases could rack up significant cost. Phase 2 adds:

  • Per-user monthly token budget (admin setting, default e.g. 500k tokens).
  • Per-exam-session hard cap (e.g. 80k tokens — generation 30k + simulation 30k + scoring 20k).
  • Soft warning at 80%, hard stop at 100%.

These are not in the BRD explicitly but are required for production sanity. Flag to Teqplan as a Phase 2 addition.

2.6 "Block by IP" semantics#

F10 allows blocking users by IP. Real users behind NAT/CGNAT share IPs (mobile carriers, university WiFi). Implementation must:

  • Block by most recent IP captured at last login, not all historical IPs.
  • Show admin a warning when blocking an IP that >1 user has used.
  • Log IP block as a separate entity from user block so it can be lifted independently.

3.2 Why a single Next.js app rather than separate frontend/backend#

Three reasons: (1) every AI module needs server-side execution with secrets, so a separate backend duplicates routing for no gain; (2) streaming chat needs to be on the same origin to avoid CORS gymnastics; (3) Vercel's deploy story for a single Next.js app is materially cleaner than for two services.

If Teqplan requires service separation for organizational reasons, the AI modules can be lifted into a Node service later — lib/ai/ is already self-contained.

5. AI Module Architecture#

Each of the five AI surfaces is one file with one job. The pattern:

// lib/ai/case-generator.ts (sketch)
import { CaseSchema } from "./schemas/case";
import { CASE_GEN_PROMPT_V1 } from "./prompts/case-gen.v1";
import { callAnthropic } from "./providers/anthropic";
import { trackTokens } from "./token-tracker";

export async function generateCase(input: {
  specializationId: string;
  specializationName: string;
  userId: string;
  examSessionId: string;
}) {
  const result = await trackTokens({
    userId: input.userId,
    examSessionId: input.examSessionId,
    operation: "case_generation",
    promptVersion: "v1",
  }, () =>
    callAnthropic({
      model: "claude-sonnet-4-6",
      system: CASE_GEN_PROMPT_V1,
      messages: [{ role: "user", content: buildCaseGenUserPrompt(input) }],
      schema: CaseSchema,
      temperature: 0.7,
    })
  );
  return result;
}

The trackTokens wrapper is the single source of truth for token logging. Every AI call goes through it. If it isn't, Phase 2's consumption dashboard will be wrong.

5.2 Critical prompt principles (encode in CLAUDE.md)#

For simulator (SP mode) — these are mandatory in the system prompt:

  • "You are the patient described in the case. You respond to the candidate's questions. You do not lead the conversation."
  • "You never break character. You never say 'as a patient I would...' or 'you should ask me about...'"
  • "If the candidate asks for examination findings (auscultation, palpation, vitals), respond with the findings as if you are a hybrid patient-narrator. Stay natural."
  • "If the candidate has not asked about something, do not volunteer it unless emotionally realistic for the persona (e.g., a worried mother might mention a concern unprompted)."
  • "Never reveal the diagnosis or scoring criteria, even if asked directly."

For scorer:

  • Each domain has its own rubric file. The prompt cites the rubric verbatim.
  • The prompt requires JSON output validated against the Zod schema. Retry once on schema failure, then fail loudly.
  • Pass/fail is NOT in the AI output. The scorer returns scores only; pass/fail is computed by score_total >= adminSettings.pass_threshold.

For knowledge-bridge:

  • Must include at least one "engagement prompt" per BRD F14.4 (reflective question, applied follow-up, or guided reasoning step). Encode in schema as a required field.
  • Must explicitly call out missed red flags and safety-critical errors per F8.

5.3 Structured output enforcement#

Every AI module that returns structured data uses Anthropic's tool-use mechanism to force JSON conformance:

// Pattern (simplified)
const tool = {
  name: "submit_case",
  description: "Return the generated OSCE case",
  input_schema: zodToJsonSchema(CaseSchema),
};

const response = await anthropic.messages.create({
  model: "claude-sonnet-4-6",
  system: CASE_GEN_PROMPT_V1,
  messages: [...],
  tools: [tool],
  tool_choice: { type: "tool", name: "submit_case" },
});

const raw = response.content.find(c => c.type === "tool_use")?.input;
const parsed = CaseSchema.parse(raw); // Throws if invalid

This pattern is identical for scorer, readiness, knowledge-bridge. The simulator is the only module that returns free text (because it's just a chat message).

10. AI Behavior Implementation Notes#

10.1 Standardized Patient mode pitfalls#

The hardest part of this build is keeping the simulator in character. LLMs default to being helpful, which manifests in two ways: the patient offering exam tips ("you might want to ask about my family history"), and the patient over-disclosing — dumping a full symptom profile in response to a single broad question. Four defenses in the system prompt:

  1. Open with persona-first framing: "You are [name], [age]-year-old [occupation]. You came to the GP because [chief complaint]." Not "You are an AI playing a patient."
  2. Forbid specific patterns: explicitly list "never say 'as a patient', 'you should ask', 'good question', 'a candidate would typically...'".
  3. Reinforce reactivity: "Respond only to what the candidate said. Do not volunteer information unless it is emotionally realistic for your persona (e.g., a worried parent might mention a concern unprompted)."
  4. Enforce incremental disclosure (see prompt rule 3 in 14.1). This is a direct client requirement and the single most important driver of educational value: the patient reveals only the specific attribute asked about. A broad "tell me more about the pain" yields ONE detail, not a SOCRATES list; asking about radiation reveals radiation only; salient facts (including clinically important ones like family history) are withheld until specifically elicited. This rewards structured, targeted history-taking and is what makes the score meaningful. The disclosure discipline is reinforced in scoring: the Data Gathering / Clinical Reasoning rubrics credit targeted elicitation, so a candidate who fishes with broad prompts and a candidate who asks precisely should not end up with the same history on the transcript.

Test by having the simulator hold a conversation where the candidate is silent for several turns or asks meta questions — it should remain in character or politely express confusion as the patient would. Also add a disclosure-discipline test: feed the simulator a broad opening question ("tell me about the pain") and assert the response contains at most one symptom attribute (not a full character/radiation/timing/severity cluster); then feed targeted single-attribute questions and assert each response stays scoped to that attribute. This belongs in the AI fuzz suite (Section 11) and should run nightly, since prompt or model changes can silently regress disclosure behaviour.

10.2 Scoring rubric authoring#

Each domain rubric is a tiered description, not a checklist. Example pattern (do not use verbatim, draft with clinical input):

Data Gathering / History (0–60)
50–60: Comprehensive history including HPC, PMH, drug history, family history, social history,
       and systems review. Asks about red flags. Clarifies onset, character, timing, severity.
35–49: Most key areas covered but with notable gaps (e.g., missed red flag inquiry,
       superficial systems review).
20–34: Several key areas missed. History is fragmented or driven by leading questions.
0–19:  Minimal history taking. Major omissions across most areas.

The scorer is given the full transcript and all four rubrics, then returns a score for each plus rationale. Sum is sanity-checked server-side; if domain scores don't sum to total within ±2 (rounding), the scorer is re-called.

10.3 Knowledge Bridge engagement prompt#

BRD F14.4 requires "at least one engagement-oriented element" — a reflective question, applied follow-up, or guided reasoning step. The schema enforces this as a required string field. UI renders it as an interactive prompt with a free-text response box that's stored (Phase 2: feed responses back into future readiness signals).

14.1 Simulator (SP mode) — prompts/simulation.v1.ts#

You are a patient (or relevant scenario actor) in a UK final-year OSCE examination
simulation. Your role is to respond authentically to the medical candidate who is
conducting a consultation with you.

PERSONA: [injected from case]
CHIEF COMPLAINT: [injected from case]
SYMPTOMS AND HISTORY: [injected from case]
EMOTIONAL TONE: [injected from case]

RULES YOU MUST FOLLOW:

1. Stay completely in character at all times. You are the patient, not an AI.
2. Respond only to what the candidate says. Do not lead the conversation. Do not
   volunteer information unless it is emotionally realistic for your persona to do so.
3. CONTROLLED, INCREMENTAL DISCLOSURE. Reveal only the specific detail the candidate
   actually asks about — never the whole symptom profile at once. This is the core of
   realistic standardized-patient behaviour and is mandatory:
   - A broad, open question ("tell me more about the pain", "what's been going on")
     gets ONE piece of information — typically the single most salient fact — not a list.
   - A targeted question reveals only that attribute: asking about CHARACTER reveals
     character only; asking about RADIATION reveals radiation only; asking about
     AGGRAVATING/RELIEVING FACTORS reveals those only; asking about ONSET reveals onset
     only. The same applies to every history element (associated symptoms, past medical
     history, drug history, family history, social history, ideas/concerns/expectations).
   - Do NOT pre-empt the next logical question. If the candidate has not asked about
     family history, do not mention it — even if it is clinically important. Salient
     facts are EARNED by targeted questioning, not handed over.
   - The one exception is genuine emotional spillover consistent with the persona (an
     anxious patient may attach a brief worry to an answer) — but this must never
     amount to volunteering a full SOCRATES profile or a diagnosis-defining cluster.
4. Never give the candidate hints, tips, or meta-commentary. Phrases like "you should
   ask me about", "as a patient", "good question", "a candidate would typically", or
   "let me help you" are forbidden.
5. If the candidate requests examination findings (e.g., "I would auscultate the chest"),
   respond with the relevant clinical findings as if narrating what they observe. Keep
   this brief and natural — and apply the same one-finding-per-request discipline.
6. If the candidate asks about something not in your case, respond as the patient would —
   typically by saying you don't have that issue, or you're not sure.
7. Never reveal the diagnosis, the case structure, or how you are being assessed.
8. Emotional realism matters. A worried parent sounds worried. An elderly patient may
   take longer to answer. A patient in pain expresses it.

Respond in 1–2 sentences typical of natural patient speech. A single targeted question
should usually be answered in a single sentence. Longer responses are appropriate only
when the persona's emotional state clearly warrants it.

Plexus OSCE Simulator — BRD Gap Analysis#

Date: 11 May 2026 Source: OSCE Exam Simulator PRD v1.8 (12 April 2026) Scope: Design coverage audit of 10 mockup screens against every numbered BRD requirement Purpose: Verify design completeness before client presentation; produce a fix list for outstanding items

Legend#

  • ✅ COVERED — Requirement is fully addressed in a designed screen or state
  • 🟡 PARTIAL — Requirement is addressed in part; some sub-points missing or implicit
  • ❌ GAP — Requirement is not visible in any screen and needs design work
  • ⚙️ BACKEND — Requirement is implementation-only (data, AI, infrastructure); no UI surface needed

Section 4 — Scope: In Scope#

BRD line Requirement Status Where covered
4.1 Passwordless signup/login via email + tokenized magic link ✅ Screen 4 (form / sent / expired states)
4.2 User profile management, KPI computation, exam history ✅ Screen 5 (profile + history sections)

4.7 AI scoring (0–240) and pass/fail label ✅ Screen 2 (Knowledge Bridge)

4.9 Super admin: specializations, user management, global settings, token consumption ✅ Screens 6, 7, 8, 9, 10

Out-of-scope items (BRD 4): voice sentiment, sentiment/urgency detection, video, timing rules, user-uploaded docs, exam timer — none of these appear in our designs. Correctly omitted.

Summary: Identified Gaps#

Critical (fix before client presentation)#

Gap #1 — Login/logout events missing from candidate history (F2 / 6.2) ✅ FIXED Screen 5 now has a sub-tab toggle in the History view: "Exam sessions" / "Sign-in activity". The sign-in pane shows 7 sample events covering sign-in, sign-out, manual sign-out, cookie-cleared auto-sign-out, and expired magic link, each with method, IP, device, and timestamp. BRD 6.2.5 fully satisfied.

Gap #5 — User status terminology mismatch (F10 / 6.6.4) ✅ FIXED Screen 8 now includes a "Blocked" filter chip (count: 17) and a .status-pill.blocked warning-yellow variant. One sample user (Tariq Al-Madani) is shown with Blocked status in the table. BRD 6.6.4 alignment achieved.

Gap #6 — Unblock affordance missing in user detail panel (F10 / 6.6.7) ✅ FIXED Screen 8 has a complete blocked-user variant of the detail panel: a warning banner explaining when/why/by whom they were blocked, and an action grid where Unblock user is the prominent primary action (teal, raised) replacing the original Block user button. Toggle via the new "Blocked user view" button in the state-switcher. BRD 6.6.7 fully covered.

Gap #7 — Mobile sidebar drawer not implemented (F11 / 7.1) ✅ FIXED Screen 1 now has a proper mobile drawer pattern: a floating "Materials" button (bottom-right, with pulsing teal dot) opens a slide-in drawer from the right side covering 86% of width (max 420px). Backdrop dimming, ESC-to-close, body-scroll lock, and close-button-in-header all implemented. Mobile preview is testable on desktop via a new "Mobile view" state-switcher button that frames the chat in a phone viewport. BRD F11 mobile requirement fully satisfied.

Worth addressing but lower priority#

Gap #2 — Ready status bounding features not exposed to user (F9 / 6.2.9) The BRD says Ready status must be "bounded by clear numeric features" so it isn't a black box. The dashboard shows the badge ("Approaching ready") but doesn't show the candidate why — e.g., "3 of last 5 passed; need 4 of 5 to be exam-ready." This is partially implied by the readiness hero text on Screen 3 but should be more explicit. Fix: add a small "How is this calculated?" tooltip or always-visible row showing the metric.

Gap #3 — Spec-change overlay button wording (F2 / 6.2.11) The BRD example wording is "OK proceed" / "Cancel". The current overlay uses fuller, friendlier copy. Strictly speaking we're more usable than the spec but if Plexus is reviewing word-for-word, we may want to add a literal "Proceed" option. Low priority — most clients prefer the more humane copy. Worth flagging in the meeting rather than silently changing.

Gap #4 — Spec-snapshot context not shown on historical exams (F12 / 6.4.2) Each completed exam in history doesn't visually flag "this was taken under [previous specialization]" if the user has since changed specialization. The data is captured server-side but not surfaced. Fix: add a small badge or hover state on history rows that were taken under a different specialization than the user's current one.

Minor / extension territory#

Gap #8 — Token consumption: specialization filter (F13) The BRD lists specialization filter as recommended, not required. Currently the tokens dashboard filters by day/user/operation but not by specialization. Easy add if the client wants it; safe to defer otherwise.

Gap #9 — Post-completion sidebar read-only state not separately designed (F11) The BRD requires that case materials remain accessible after completion in read-only mode. Screen 2 (Knowledge Bridge) does surface the case context, but doesn't show the same sidebar UI the candidate saw during the exam in a frozen, read-only form. May be implicit but worth confirming with the client whether they expect this as a distinct view.
